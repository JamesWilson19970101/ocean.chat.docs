Ocean Chat Authentication Architecture

Authentication serves as the foundational security layer for Ocean Chat, gatekeeping access for millions of concurrent users. Designing an authentication system for an Instant Messaging (IM) platform of this scale requires balancing extreme performance with rigorous security constraints.

This page explains the overarching architecture of the Ocean Chat Authentication Service, detailing how it orchestrates the Zero-I/O verification pattern with a JWT Hybrid management strategy to achieve these goals.

The Context: Balancing Scale and Security​

In traditional web applications, authentication often relies on centralized sessions or stateful token validation. However, Ocean Chat is designed to handle 10 million concurrent connections. At this scale, traditional stateful verification creates a massive bottleneck. Every request verifying a token against a central database (like Redis) generates network overhead, potentially leading to cascading infrastructure failures under heavy load.

Conversely, purely stateless tokens (like standard long-lived JWTs) pose unacceptable security risks. If a token is stolen, it cannot be revoked without changing the signing key for all users. The challenge for Ocean Chat's Auth Service is to deliver the performance of stateless tokens while retaining the security control of stateful sessions.

Core Concept: The Decentralized Verifier Model​

The Ocean Chat Authentication Service resolves this tension by adopting a Centralized Issuer, Decentralized Verifier architecture.

The Auth Service acts as the sole Centralized Issuer. It is the only component in the infrastructure holding the private cryptographic keys necessary to generate valid user credentials.

The API Gateways and WebSocket nodes act as Decentralized Verifiers. They operate entirely independently of the Auth Service for routine traffic. To make this possible without compromising security, the Auth Service delegates responsibilities using two core mechanisms:

  1. Zero-I/O Authentication: The gateways use an asymmetric public key (RS256) to cryptographically verify tokens locally. They maintain an event-driven, in-memory blacklist to handle token revocations instantly, entirely eliminating the need for remote database queries during request validation.
  2. JWT Hybrid Strategy: The Auth Service issues two distinct tokens: a short-lived Access Token (stored purely in memory) for high-frequency, stateless API access, and a long-lived Refresh Token (stored in a secure HttpOnly cookie) for low-frequency, stateful session renewal.

Alternatives and Trade-offs​

When designing this architecture, several alternatives were evaluated:

  • Stateful Sessions (Redis/Memcached): While providing perfect consistency and easy revocation, the network I/O required for every request is fundamentally incompatible with our 10M concurrent user target.
  • Standard Stateless JWTs: Offers excellent scalability by eliminating database lookups, but sacrifices security. A compromised token remains valid until it expires, offering no defense against active attacks.

Ocean Chat's architecture is a deliberate compromise. We trade microscopic moments of consistency (due to the asynchronous nature of the revocation event bus) to achieve infinite horizontal scalability and robust session control.

Higher-Level Perspective​

The Auth Service is more than just a token generator; it is the orchestrator of trust across the entire Ocean Chat ecosystem. By isolating the issuance of trust to a highly secure central node and distributing the verification of trust to the highly scalable edge, Ocean Chat ensures that the front door remains both secure against compromise and completely frictionless for legitimate users.
