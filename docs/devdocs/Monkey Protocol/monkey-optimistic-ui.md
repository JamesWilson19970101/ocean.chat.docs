# Auxiliary Design for Optimistic UI

In C-end Instant Messaging (IM) products, the ultimate user experience is **"never feeling any network latency."** When a user clicks the "Send" button, the message bubble should appear on the chat screen within 1 millisecond, rather than waiting several hundred milliseconds for a network round-trip (RTT) to complete.

This "render first, confirm later" mechanism is known as **Optimistic UI**. Since Monkey Protocol uses a strict chronological model with `SyncSeqId` assigned uniformly by the server, the client faces challenges like "unknown ID" and "state loops" during optimistic rendering. This document specifies how the client SDK leverages the underlying design of Monkey Protocol to perfectly close the loop for Optimistic UI rendering and error prevention.

---

## 1. Core Link: `ClientMsgId` (Client Deduplication and Association Anchor)

In Monkey Protocol, `SyncSeqId` (a 64-bit long integer) is the absolute authoritative message cursor in a session and is monotonically assigned by the `oceanchat-message` service. This means that at the moment a client sends a message, it **absolutely does not know** the final `SyncSeqId` of that message.

The foundation of Optimistic UI lies in **`ClientMsgId` (strictly a time-ordered UUIDv7)**:

1. **Generation Before Sending**: When the UI layer triggers a send action, the client SDK immediately generates a globally unique `ClientMsgId`.
2. **Dual Identity**:
   - **For the Server**: It is the unique credential for achieving high-concurrency idempotency (written to Redis to prevent duplicate storage caused by network reconnection retries).
   - **For the Client**: It is the **unique anchor** connecting the "local fake state" with the "server real state."

---

## 2. State Machine Design for Client Local Storage (Local DB)

To support Optimistic UI, the message table (`Message`) in the client's local database (IndexedDB for Web, SQLite for Mobile) must have the following three key fields:

- `client_msg_id`: (String, Unique Index) Generated by the client.
- `sync_seq_id`: (Int64, Nullable) Generated by the server.
- `send_status`: (Enum) Local sending status, including `SENDING`, `SENT`, and `FAILED`.

### 2.1 State Transition: The Moment of Sending (Optimistic Render)

When a user clicks send:

1. The UI assembles rich text or text data and calls the SDK.
2. The SDK generates `ClientMsgId = "uuidv7-1234"`.
3. The SDK **immediately** writes the message to the local DB:
   - `client_msg_id = "uuidv7-1234"`
   - `sync_seq_id = null` (or set to a very large local fake ID to ensure it sorts at the bottom)
   - `send_status = SENDING`
   - `created_at = Date.now()`
4. **UI Response**: Changes in the local database are immediately pushed to the interface via reactive streams (e.g., RxJS, Flow, LiveData). A message bubble appears at the bottom of the screen, accompanied by a "loading/spinning" icon.

## 3. Protocol Loop: 3 Paths for State Correction

After a message is sent via Monkey Protocol's `[0x05] MSG_UP`, the client encounters three different protocol loop scenarios. The SDK must accurately intercept and correct the state for these scenarios.

### Path A: The Perfect Path (Receipt of `MSG_UP_ACK`)

The server processes the request rapidly, passes the write barrier, and sends `[0x06] MSG_UP_ACK` back via the original TCP/WS connection.

1. **ReqId Matching**: The client's underlying interceptor uses the `ReqId` in the 12-byte Header to find which underlying network request was just sent.
2. **Obtain Server State**: The Payload of `MSG_UP_ACK` carries the official `SyncSeqId` (e.g., `1005`) assigned by the server and the server's `ServerTimestamp`.
3. **Update DB (State Finalization)**: The SDK finds the `ClientMsgId` based on the `ReqId` and updates the local database:
   - `send_status -> SENT`
   - `sync_seq_id -> 1005`
4. **UI Response**: The "loading" icon disappears, and the optimistic render loop is perfectly closed.

### Path B: Edge Case Path (ACK Lost, but Recovered via `MSG_NOTIFY`)

In extremely weak network conditions, the `MSG_UP` sent by the client successfully reaches the server and is stored, but the network disconnects just as the server returns `MSG_UP_ACK`.
At this point, the client's optimistic UI is still stuck in the `SENDING` state.

1. **Reconnection and Implicit Pull**: After the network recovers, the client automatically reconnects. If there are new messages in the group, the server sends `[0x08] MSG_NOTIFY`, and the client triggers an incremental pull via an HTTP short connection (`Sync`).
2. **Encountering Its Own Message**: In the incremental message list returned by the HTTP interface, the client iterates and finds the message entity it just sent.
3. **ClientMsgId Fusion Interception**: Before inserting new messages into the DB, the SDK must perform a strong validation:
   - Check if `Message.client_msg_id` already exists in the local DB.
   - If it exists and `send_status == SENDING`, it means **this is the message I just sent whose ACK was lost.**
4. **Update DB (State Finalization)**: The SDK does not insert a new row but instead `UPDATE`s the local fake message in the `SENDING` state, overwriting its `sync_seq_id` with the real ID and setting `send_status` to `SENT`.

### Path C: Exception Interception Path (Receipt of `EXCEPTION_ACK`)

Due to sensitive words, being blocked by the other party, or a token expiring, the gateway actively sends `[0x0C] EXCEPTION_ACK`.

1. **ReqId Matching**: Similarly, use the `ReqId` in the Header to find the corresponding request interceptor.
2. **Update DB (State Failed)**:
   - `send_status -> FAILED`
3. **UI Response**: The "loading" icon turns into a red exclamation mark `!`. Clicking the exclamation mark can show the specific error reason returned by the server (`ExceptionAck.message`).

## 4. Deep Synergy with the "In-Flight Queue"

The previously designed **automatic replay mechanism for the In-Flight Queue during disconnection** is a perfect match for Optimistic UI.

When a user enters a subway (loses connection) and sends a message:

1. The UI performs optimistic rendering with a state of `SENDING`.
2. The underlying network discovers the dead link, and the message is stored in the memory's `In-Flight Queue`.
3. The user switches out of the App or turns off the screen. As long as the process isn't killed, the message waits silently in the queue.
4. **UI Behavior**: During this period, no matter how much time passes, the UI remains in `SENDING` (or changes to something like "Waiting for network..." after a timeout). **It must not be easily marked as `FAILED`** because the underlying layer has taken over the replay responsibility.
5. After the network recovers, the `In-Flight Queue` automatically sends the packet with the original `ReqId` and `ClientMsgId`. Upon receiving the ACK, the UI's `SENDING` state is finalized. This extreme fault tolerance gives users great peace of mind.

## 5. Advanced: Optimistic Entity Allocation for Complex Business Entities

Optimistic UI is not limited to text messages; it can also be applied to complex interactions like "creating a group chat" or "sending multiple images."

Monkey Protocol advocates for **"the client as the pre-allocator of IDs"** (using hashes or UUIDv7s for dimensionality reduction):

- **Sending a Large Video**:
  1. Before the video starts uploading to OSS, the SDK pre-generates a `ClientMsgId` and inserts it into the local DB.
  2. The UI layer uses this fake data to directly render a video bubble with `0%` progress in the conversation list.
  3. After the HTTP upload is complete and the long-connection `MSG_UP` control plane notification is received, the state changes from `SENDING` to `SENT`.
- **Temporary Sorting Weights**:
  Since optimistic messages don't have a real `SyncSeqId`, how do I ensure they appear at the very bottom of the conversation when querying the local DB?
  **Best Practice**: Use logic like `ORDER BY COALESCE(sync_seq_id, 999999999999999) + created_at ASC` in SQLite queries. Using `COALESCE` gives optimistic messages without an official ID an extremely high virtual weight, ensuring they always stick to the end of the current chat stream.

---

## 6. Summary

Optimistic UI based on Monkey Protocol is not haphazard "frontend deception" but a set of precisely coordinated data loops:

1. **`ClientMsgId` is the core anchor**, used to associate pending server data with local temporary data.
2. The **`SENDING` state** in the local database is a transition state bridging the long-connection control plane (`MSG_UP_ACK`) and the short-connection data plane (`Sync`).
3. Combined with **In-Flight Queue replay** and **server-side idempotent deduplication**, I achieve "Fire and Forget" in weak network environments, ensuring that duplicate bubbles or ghost messages are never produced.
