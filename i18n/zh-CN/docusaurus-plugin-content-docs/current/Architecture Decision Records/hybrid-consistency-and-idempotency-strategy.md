# 架构决策记录：混合一致性与幂等性策略

## 状态

**已通过 (Accepted)**

## 日期

2026-05-24

## 背景 (Context)

在像 Ocean Chat 这样的大规模分布式 IM 系统中，不同的业务场景对一致性和性能有着完全不同的需求。如果采用“一刀切”的方法——例如对所有业务都使用分布式事务，或者所有地方都采用最终一致性模式——要么会导致高频路径下不可接受的延迟，要么会增加简单管理任务的过度开发复杂度。

我需要一种策略，在保证核心操作数据完整性的同时，优化高频事件的用户体验（低延迟）。

## 决策 (Decision)

Ocean Chat 将根据具体的业务影响和频率，采用分层的混合一致性模型：

### 1. 热点业务路径（如：发消息、状态同步）

**策略：** 彻底抛弃多文档事务。

- **机制：** 利用数据库单行/单文档原子操作（如 MongoDB 的 `$inc`, `$set`）配合消息队列 (MQ) 异步派发。
- **一致性模型：** 最终一致性 (Eventual Consistency)。
- **强制要求：**
  - **全链路幂等性 (Idempotency)：** 每一次流转和状态更新必须拥有唯一的 `IdempotencyKey`。系统必须保证无论重试多少次，结果都与执行一次完全一致。
  - **安全网 (Safety Nets)：** 引入死信队列 (DLQ) 处理失败操作，并建立离线对账与补偿机制 (Reconciliation & Compensation) 来解决极端情况下的数据差异。

### 2. 冷门核心业务（如：建团队、注册）

**策略：** 拥抱 MongoDB 原生多文档事务。

- **机制：** 使用 MongoDB 标准的 ACID 事务。
- **一致性模型：** 强一致性 (Strong Consistency)。
- **理由：** 这些操作属于低频操作。使用原生事务可以用最低的开发成本换取最高的强一致性，确保基础实体的完整性。

### 3. 高频资金/库存业务（未来扩展）

**策略：** 内存原子操作 + 异步持久化。

- **机制：** 使用 Redis Lua 脚本进行内存层面的原子扣减（例如防止超卖），随后通过 MQ 驱动异步记账和数据库落库。
- **要求：** MQ 消费者端必须实现严格的防重逻辑。

### 4. 分布式基石原则

- **幂等消费者：** 所有异步消息消费者必须在设计上实现幂等性。
- **兜底保障：** 所有的最终一致性链路必须拥有死信队列或对账补偿机制作为安全底座。

## 理由 (Rationale)

- **性能：** 移除消息发送路径中的事务开销，对于在大规模场景下保持 100ms 以下的延迟至关重要。
- **开发效率：** 在“创建群组”等冷路径中使用事务，可以简化代码，无需编写复杂的自定义状态机或手动回滚逻辑。
- **可靠性：** 通过强制执行 `IdempotencyKey` 和 DLQ 模式，系统在面对网络分区或服务重启时能够保持强韧，不会产生永久性的数据损坏或业务死锁。

## 备选方案 (Alternatives Considered)

### 全局分布式事务 (2PC/Saga)

- **优点：** 统一的一致性模型。
- **缺点：** 2PC 对于 IM 消息发送来说太慢；Saga 为简单操作引入了巨大的状态管理开销。
- **结论：** 热点路径上的延迟影响是用户体验无法接受的。

### 纯粹的最终一致性

- **优点：** 极高的可扩展性。
- **缺点：** 对简单任务过度设计。为一个简单的“用户注册”构建补偿流程会增加不必要的开发周期。
- **结论：** 对于低频核心数据，强一致性更便宜且更安全。

## 后果 (Consequences)

- 热点路径的业务逻辑变得更加复杂，需要显式的幂等性检查。
- 开发人员在设计阶段必须明确区分“热点”和“冷门”业务路径。
- 由于需要监控 DLQ 并运行定期对账脚本，运维开销略有增加。
- 数据完整性由数据库锁保障转变为通过幂等性实现的“数学上保证”。

## 延伸探讨 (Extended Discussion)：高频业务的“既快又稳”防护体系

针对上述决策中提到的**“Redis 内存原子操作 + MQ 异步落库”**这种追求极致吞吐的高频业务策略，系统在整体设计哲学上必须配套极其严密的立体防护网，以弥补这一“险棋”带来的固有风险：

- **自动化补偿与对账 (Reconciliation & Compensation)：** 异步记账的本质是最终一致性。系统不仅依赖 MQ 的正向重试落库，还需要应对底层 DB 坚决拒绝落库时的逆向补偿（冲正并下发失败通知）。此外，后台必须配置定时自动对账任务，比对 Redis 内存流水与底层数据库流水，自动发现并平账错漏。
- **监控与死信告警 (DLQ & Alerting)：** 资金与核心状态业务绝不能静默失败。若 MQ 消费者反复重试落库依然失败，消息将移入死信队列 (DLQ)。DLQ 的触发必须绑定最高级别的告警通道，立刻呼叫值班工程师介入并要求在严格的 MTTR（平均恢复时间）内解决。
- **自动化优先，人工兜底 (Automation First, Human as Last Defense)：** 现代系统 99.9% 的异常（如网络闪断、短时限流）依靠自动重试与对账自愈。只有在自动化机制全部失效的极端情况（如严重的代码逻辑 Bug 产生无法落库的“毒消息”或脏数据）下，才由人工介入排查，修复 Bug 后通过管理后台一键重放或人工调账进行兜底。
- **不信任硬件的基石与容灾 (Infrastructure Fault Tolerance)：** 考虑到 Redis 内存数据在整机断电瞬间极易丢失的特性，系统除了依赖硬件级保障（如机房 UPS 不间断电源、企业级 SSD 的掉电保护电容）外，更要在软件架构上秉持“随时可能被拔电源”的云原生哲学。通过 Redis 集群主从多副本 (Replication) 与高可用哨兵，确保故障机宕机后从节点在极短时间内自动晋升接管业务，配合对账系统找回断电瞬间的毫秒级数据差异。
