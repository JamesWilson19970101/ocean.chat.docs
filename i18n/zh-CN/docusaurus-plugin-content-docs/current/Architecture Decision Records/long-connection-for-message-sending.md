# ADR: 采用长连接实现消息发送

## 状态

已接受

## 日期

2026-05-20

## 背景 (Context)

在现代 IM 系统中，客户端需要高效地将消息传输到服务端。目前主要有两种传输策略：

1. **短连接 (HTTP/REST)：** 客户端为每条消息发起一个新的 HTTP 请求，发送数据载荷后关闭连接（或利用 keep-alive，但这仍会产生请求开销）。
2. **长连接 (WebSocket/TCP)：** 客户端与服务端建立一个持久的双工连接。所有实时的消息发送和接收均在这个已建立的单一通道上进行。

## 决策 (Decision)

采用 **长连接 (WebSocket/TCP)** 作为客户端向服务端发送消息的主要传输机制。

## 考虑的替代方案 (Alternatives Considered)

### 短连接 (HTTP API)

- **优势：**
  - 实现简单且天然无状态。
  - 能够利用现有的 HTTP 基础设施（API 网关、标准负载均衡器）。
  - 内置的请求/响应语义简化了客户端的错误处理。
- **劣势：**
  - **协议开销：** 每个请求都需要携带 HTTP 头部。如果没有持久的 keep-alive，还会产生 TCP/TLS 握手的时间惩罚，显著增加延迟。
  - **功耗：** 频繁唤醒移动端无线电天线来建立出站连接，会严重消耗移动设备的电池。
  - **延迟不一致：** 在空闲一段时间后发送消息时，初始连接时间较慢。
- **被拒绝：** 对移动端电池续航的严重影响以及不必要的协议开销，使其不适合高频聊天的应用场景。

### 长连接 (WebSocket / TCP)

- **优势：**
  - **省电 (Power Efficiency)：** 维持单一持久连接可以将移动设备无线电唤醒状态降至最低，从而大幅节省电池电量。
  - **高性能 (High Performance)：** 消除了重复的 TCP/TLS 握手和 HTTP 头部开销。消息采用最小字节的帧格式，提供超低延迟。
  - **双工优化：** 复用已建立的通道，既可用于发送出站消息，也可用于接收入站的实时事件（例如：推送消息、正在输入状态）。
- **劣势：**
  - **有状态的复杂性：** 需要健壮的连接生命周期管理，包括应用层心跳（ping/pong）和重连逻辑。
  - **基础设施要求：** 网关负载均衡器必须配置为支持长时间保持的连接以及代理 WebSocket/TCP 流量。
- **被接受：** 省电和延迟改善带来的收益大于基础设施的复杂性。

## 后果 (Consequences)

- **客户端实现：** 客户端 SDK 必须实现健壮的连接管理。这包括用于重连的指数退避算法，以及用于检测半开或死连接的应用层心跳机制。
- **服务端架构：** 接入/网关层必须设计为能够并发保持数十万个持久连接（利用 epoll/kqueue），并能将入站消息快速路由到后端服务的有状态节点。
- **大载荷回退策略：** 虽然长连接处理实时文本和较小的数据载荷，但会保留 HTTP REST 回退机制，用于上传大型多媒体文件（图片、视频），以避免阻塞主要的双工数据流。

:::tip 性能提示
通过集中现有的长连接通道进行消息分发，系统避免了后续消息发送时经典的 TCP 慢启动阶段，确保即使在短暂空闲后也能实现瞬间送达。
:::
