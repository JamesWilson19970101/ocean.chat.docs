# ADR-005: 全面采用 `consume()` 处理 NATS JetStream 消费者

## 状态

已接受 (Accepted)

## 日期

2024-05-15

## 背景 (Context)

Ocean Chat 深度依赖 NATS JetStream 来实现异步消息投递、事件溯源以及维护预写日志 (WAL)。为了支撑十万级并发连接，微服务必须高效地从 NATS 消费消息，且在流量洪峰（例如万人大群广播或服务器冷启动）期间不能被压垮。

NATS JetStream 提供了多种消费消息的方式：

- **推送消费者 / Push Consumers (`subscribe()`)**：服务端以最快的速度向客户端主动推送消息。
- **拉取消费者 / Pull Consumers (`consume()`, `fetch()`)**：客户端根据自身的处理能力，显式地向服务端请求批量消息。
- **有序消费者 / Ordered Consumers**：一种简化的临时消费者，自动处理序列追踪，但放弃了显式的消息确认 (ACK)。

为了保证系统的绝对稳定、防止内存溢出 (OOM) 崩溃，并简化开发体验，我需要在 Ocean Chat 的所有微服务中确立一个标准化的消息消费模式。

## 决策 (Decision)

在 Ocean Chat 生态系统中，对于所有需要持久化和高吞吐量的 NATS JetStream 消息消费，**全面采用拉取消费者 (Pull Consumers)** 并标准化使用 **`consume()`** API。

简而言之：系统全部选择 `consume()`，主要还是因为全部选择 `consume()` 是完全够用的，且能提供最高级别的安全感。

## `consume()` 的底层原理 (推拉结合机制)

在底层，`consume()` 实现了一种智能的推拉结合机制（具体来说是带有批量预取的长轮询 / Long Polling）。

- **消息获取过程:** 当初始化 `consume()` 时，JetStream 客户端并不是一条一条地要消息。它会向服务端发送一个拉取请求，要求获取一批消息（数量由 `max_messages` 控制）。
- **服务端推送:** NATS 服务端收到此请求后，会立即开始向客户端“推送”可用消息，直到达到请求的批量大小。
- **超时处理:** 每个拉取请求都有一个过期时间（`expires` 参数）。如果在计时器耗尽之前服务端没有新消息可发，该拉取请求会优雅地超时失效。
- **重新获取的时机:** 客户端库在底层自动管理这个生命周期。一旦某个拉取请求超时失效（或者已经消费的消息超出阈值），客户端会**立即且透明地**在后台发起一个新的拉取请求。这保证了客户端的本地缓冲区始终有消息储备，供 `for await (const msg of iter)` 循环消费，从而将延迟降到最低。

## 容量证明：为何对 100k+ 并发量是完全够用的

一个常见的误区是：与纯推送相比，基于拉取的消费者对于高并发来说太慢了。然而，对于 Ocean Chat 设定的十万级 (100,000+) 并发连接目标，使用 `consume()` 在数学和工程实践上都是完全够用的：

1. **并发连接不等于吞吐量 (TPS):** 10万个并发的 WebSocket 连接并不意味着每秒产生 10万条消息。在典型的 IM 系统中，同一秒内只有极小比例的用户在发送消息。一个 10万并发的系统，其常规吞吐量通常在 1,000 到 5,000 TPS 之间。
2. **批量拉取的效率:** `consume()` 并不是一条一条去拉取消息的。通过预取批次（例如一次请求 100 或 500 条消息），拉取请求的网络开销被数百条消息平摊了。NATS 处理这种请求的延迟几乎为零。
3. **Node.js 的处理能力:** 假设只是标准的 JSON 解析和异步数据库写入，单个使用 `consume()` 的 Node.js 实例可以轻松处理 5,000+ TPS 的业务逻辑。
4. **无痛水平扩展:** 由于 `consume()` 自带背压机制，我们可以极其安全地进行水平扩展。如果吞吐量突增到 20,000 TPS，只需在队列组 (Queue Group) 中启动 4-5 个服务实例即可。它们会平滑地拉取消息，且绝不会压垮任何单一实例。

因此，拉取请求所谓的“性能损耗”在实际场景中完全可以忽略不计；
相反，它带来的稳定性保障（彻底告别 OOM）使其成为十万级并发架构的绝对最优解。

## 考虑过的替代方案 (Alternatives Considered)

### 推送消费者 (`subscribe()`)

- **优点 (Pros):** 概念简单；传统 Pub/Sub 模型中常用的方式。对于极低流量的流，延迟略低。
- **缺点 (Cons):** 极易遭遇“慢消费者 (Slow Consumer)”问题。如果下游服务（如 MongoDB 落库）变慢，NATS 依然会源源不断地把消息强塞进 Node.js 的事件循环中。这会导致缓冲区无限增长、内存耗尽，最终导致进程 OOM 崩溃。
- **拒绝原因:** 缺乏隐式的背压 (Backpressure) 机制，完全不适合处理 IM 系统中突发的高流量洪峰。

### 有序消费者 (Ordered Consumers)

- **优点 (Pros):** 如果检测到序列号空洞，会自动重建消费者，速度极快。
- **缺点 (Cons):** 严格作为临时消费者 (Ephemeral) 运行，且不支持显式消息确认 (`AckNone`)。
- **拒绝原因:** Ocean Chat 需要绝对的消息可靠性（至少投递一次 / At-Least-Once）。缺乏显式 ACK 意味着如果进程在处理消息时崩溃，将导致数据的永久丢失。

### 使用 `fetch()` 的拉取消费者

- **优点 (Pros):** 基于拉取，提供极佳的流量控制。
- **缺点 (Cons):** `fetch()` 是一种离散的请求-响应机制（即“现在给我 N 条消息”）。对于连续的流处理来说效率较低，因为需要不断发起拉取请求，增加了批次之间的网络延迟。
- **拒绝原因 (优先选择 `consume()`):** 虽然 `fetch()` 适用于一次性任务，但 `consume()` 提供了一个连续的、异步可迭代 (async-iterable) 的流，它在底层自动管理拉取请求（预取），同时维持了严格的流量控制。

## 后果 (Consequences)

- **内建背压 (Built-in Backpressure):** 通过使用 `consume()`，应用程序可以在原生层面控制处理速率。服务在内存中缓冲的消息绝不会超过其处理能力，从根本上消除了由 NATS 引发的 OOM 崩溃。
- **水平扩展 (Horizontal Scalability):** 多个服务实例可以安全地绑定到同一个持久化名称 (Durable Name) 上，形成队列组 (Queue Group)，无缝地分发工作负载。
- **极致简化 (Simplicity):** 标准化采用单一方法降低了认知负荷。开发人员清楚 `consume()` 是系统中几乎所有用例的正确、安全之选，因为它“足够好用且万能”。
- **开发约束:** 开发人员必须确保在成功处理消息后显式调用确认（`msg.ack()`），因为 `consume()` 默认采用 `AckExplicit`（显式确认）策略。
