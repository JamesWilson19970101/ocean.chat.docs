# JWT 混合认证

在现代 Web 应用和微服务架构中，身份认证是安全体系的基石。基于 JWT (JSON Web Token) 的认证因其无状态、可扩展和跨域友好的特性而被广泛采用。然而，单一的、长有效期的 JWT 令牌在安全性和灵活性上存在固有风险。

:::tip 核心思想
本平台采用 **JWT 混合模式认证策略**。该策略通过结合使用短期的 `Access Token` (访问令牌) 和长期的 `Refresh Token` (刷新令牌)，并在每一个环节实施纵深防御，以期在高性能、高安全性和优秀的用户体验之间取得最佳平衡。
:::

## Access Token (AT - 访问令牌)

- 角色: 客户端访问受保护资源的“通行证”。它是高频使用的令牌。

- 特性:

  - **生命周期短**: 有效期极短，计划设置为 **5 到 15 分钟**。
  - **无状态验证**: 服务器仅验证其签名和有效期即可信任其内容，无需查询数据库。这保证了极高的处理性能。
  - **风险**: 由于其高频使用和传输，被盗风险相对较高。其短暂的生命周期是缓解此风险的关键。

- 载荷 (Payload): 包含用户的非敏感身份信息，如用户 ID、角色等，遵循最小权限原则。

## Refresh Token (RT - 刷新令牌)

- 角色: 获取新的 Access Token 的唯一凭证，可以看作是维持用户登录状态的“会话钥匙”。它是低频使用的令牌。

- 特性:
  - **生命周期长**: 有效期较长，计划设置为 **7 到 30 天**。
  - **有状态验证**: 服务器在处理 RT 时，必须查询数据库或缓存，以验证其是否仍然有效。这提供了撤销会话的能力。
  - **风险**: 一旦泄露，将导致严重的安全后果。因此，必须对其进行最严格的存储和传输保护。

## 混合模式

该模式的核心思想是将认证职责分离：

- 日常访问 (99% 的场景): 使用无状态、高性能的 Access Token。

- 会话续期 (1% 的场景): 使用有状态、高安全的 Refresh Token。

这种设计使得系统在绝大多数情况下都能快速响应，同时保留了在关键时刻（如刷新会话、登出）进行严格安全检查的能力。

## 安全策略

### 安全的令牌生成与配置

:::danger 隔离密钥
Access Token 和 Refresh Token **必须**使用两个完全不同且足够复杂的密钥 (`JWT_ACCESS_SECRET` 和 `JWT_REFRESH_SECRET`) 进行签名，防止通过一个密钥推断出另一个。
:::

**最小化载荷**: Access Token 的载荷中**严禁**包含任何敏感信息（如密码、手机号、权限详情等）。只应包含必要的身份标识符（如 `userId`）和角色。

**合理的有效期**:

AT: `15m` (15 分钟)。

RT: `7d` (7 天)。

### 安全的客户端存储

#### Access Token

**首选: 存储在 JavaScript 的内存变量中**。这是最安全的方式，因为它不会在浏览器会话之间持久化，且完全免疫 CSRF 攻击。

:::tip 优点

- **极高的安全性**: 这是最能抵御 XSS (跨站脚本) 攻击的方式。因为令牌存储在一个 JavaScript 变量中，攻击者即使能注入脚本，也无法像访问 `sessionStorage` 或 `localStorage` 那样直接、持久地读取到它。
- **无 CSRF 风险**: 由于令牌需要通过 JavaScript 手动添加到 `Authorization` 请求头中，浏览器不会自动携带它，因此完全免疫 CSRF (跨站请求伪造) 攻击。
- **攻击窗口极小**: 一旦用户关闭标签页或刷新页面，内存变量就会被清空，令牌立即消失。这意味着即使令牌在某个瞬间被窃取，它的生命周期也极其短暂。
  :::

:::caution 缺点

- **用户体验较差**: 这是最大的短板。每次用户刷新页面或打开一个新的标签页访问您的应用时，内存中的 Access Token 都会丢失。这将导致应用必须立即使用 Refresh Token 去换取新的 Access Token，增加了 `/refresh` 接口的请求频率。如果网络状况不佳，用户可能会感到明显的“重新加载”或“卡顿”。
  :::

**次选: 存储在 `sessionStorage` 中**。它会在页面刷新时保留，但在浏览器窗口关闭时清除。

:::tip 优点

- **更好的用户体验**: 它可以在同一个标签页的会话中持久存在。用户刷新页面后，令牌依然有效，应用可以立即恢复状态，无需再次发起刷新令牌的请求。
- **隔离性**: `sessionStorage` 是基于标签页的，关闭标签页后数据即被清除，不会像 `localStorage` 那样永久存储。
  :::

:::caution 缺点

- **易受 XSS 攻击**: 这是它和内存存储相比最大的安全隐患。如果您的网站存在 XSS 漏洞，攻击者注入的脚本可以通过 `sessionStorage.getItem('accessToken')` 轻松地读取到令牌，然后发送到他们自己的服务器。
  :::

#### Refresh Token

**存储在浏览器的 `HttpOnly` Cookie 中。**

:::info Cookie 属性

- `HttpOnly`: 此标记禁止任何客户端 JavaScript 访问该 Cookie，从根本上杜绝了 XSS 攻击窃取 RT 的可能性。
- `Secure`: 此标记强制浏览器仅在 HTTPS 连接下发送该 Cookie。
- `SameSite=Strict` 或 `Lax`: 此标记有助于抵御 CSRF (跨站请求伪造) 攻击。
  :::

### 安全的令牌传输

**强制 HTTPS**: 所有与认证相关的端点以及整个应用，都**必须**强制使用 HTTPS (TLS) 加密。这可以防止令牌在传输过程中被中间人攻击 (MITM) 截获。

### 安全的令牌轮换与验证

**令牌轮换 (Refresh Token Rotation)**: 这是检测 RT 盗窃的核心机制。

1.  当客户端使用一个 RT (`RT_A`) 来刷新时，服务器在验证通过后，会同时返回一个新的 AT (`AT_B`) 和一个全新的 RT (`RT_B`)。
2.  服务器会立即将旧的 `RT_A` 标记为已使用或作废。

:::danger 检测到重放攻击
如果服务器之后再次收到了 `RT_A` 的刷新请求，就意味着这个令牌可能已被盗用（攻击者在重放旧令牌）。此时，服务器应立即将该用户的所有会话全部作废，强制其在所有设备上重新登录。
:::

### 安全的会话终止

**用户登出**: 用户登出时，客户端应主动请求 `/logout` 端点。服务器收到请求后，必须将数据库中存储的对应 RT 哈希值清除，从而使该 RT 立即失效。

**强制所有设备登出**: 提供一个“安全设置”选项，允许用户清除其账户下关联的所有 RT 记录，实现一键下线。

## 工作流程

步骤 1: 初始登录
客户端: 用户在登录页面输入用户名和密码，点击登录。

服务器:

1.  验证凭证是否正确。
2.  若正确，生成一对令牌：
    - 短期的 Access Token (`AT_A`)。
    - 长期的 Refresh Token (`RT_A`)。
3.  对 `RT_A` 进行哈希处理，并将哈希值 (`Hashed_RT_A`) 存储在用户数据库的对应记录中。
4.  通过 HTTP 响应将令牌发回客户端：
    - `AT_A` 放在响应体 (JSON Body) 中。
    - `RT_A` 放在 `Set-Cookie` 响应头中，并附带 `HttpOnly; Secure; SameSite=Strict` 标志。

步骤 2: 访问受保护资源
客户端:

1.  将从响应体中获取的 `AT_A` 存储在内存中。
2.  在每次向受保护 API 发送请求时，将 `AT_A` 放在 `Authorization: Bearer <AT_A>` 请求头中。

服务器:

1.  接收到请求，提取 `AT_A`。
2.  进行快速的无状态验证（检查签名和有效期）。
3.  验证通过，处理请求并返回数据。

步骤 3: Access Token 过期与刷新
客户端: 使用已过期的 `AT_A` 请求 API。

服务器: 验证 `AT_A` 发现其已过期，拒绝请求，返回 `401 Unauthorized` 状态码。

客户端 (请求拦截器):

1.  自动捕获 `401` 错误。
2.  自动向服务器的 `/auth/refresh` 端点发起一个请求（服务器设置 Cookie 时使用的一个非常重要的属性：Path）。浏览器会自动携带存储在 `HttpOnly` Cookie 中的 `RT_A`。

服务器 (`/auth/refresh` 端点):

1.  从 Cookie 中获取 `RT_A`。
2.  进行**有状态验证**:
    - 检查 `RT_A` 的签名和有效期。
    - 从 `RT_A` 的载荷中获取 `userId`。
    - 根据 `userId` 查询数据库，获取存储的 `Hashed_RT_A`。
    - 比对 `RT_A` 的哈希值与 `Hashed_RT_A` 是否匹配。
3.  **验证成功 (令牌轮换)**:
    - 生成一对全新的令牌：`AT_B` 和 `RT_B`。
    - 对 `RT_B` 进行哈希，用新的 `Hashed_RT_B` 覆盖数据库中的 `Hashed_RT_A`。
    - 将 `AT_B` 和新的 `RT_B` (通过 `HttpOnly` Cookie) 返回给客户端。
4.  **验证失败**: 返回 `403 Forbidden`，表示会话无效。

客户端 (请求拦截器):

1.  收到新的 `AT_B` 后，更新内存中的 Access Token。
2.  使用新的 `AT_B` 自动重试刚才因 `401` 失败的那个 API 请求。
3.  如果收到 `403`，则清除所有本地状态，将用户重定向到登录页面。

步骤 4: 登出
客户端: 用户点击登出按钮，向 /auth/logout 端点发送请求。

服务器:

1.  通过 Access Token 识别用户身份。
2.  在数据库中找到该用户的记录，将其 `hashedRefreshToken` 字段置为 `null`。
3.  在响应中发送一个用于清除 `HttpOnly` Cookie 的 `Set-Cookie` 头。
