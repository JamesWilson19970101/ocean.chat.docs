# 下一阶段需要处理的事项

## 1. 技术债务

- 服务健康检查监测机制。
- mongodb \_id 统一使用uuid v7 ，要 string 类型。
- 计算在线和离线用户：主节点分片（从redis中逐批读取数据，将数据扔到队列中） + MQ 消费者组负载均衡（接收数据） + Worker 节点并发计算（使用 n 个worker节点计算状态）
- 目前系统中通过幂等建实现了幂等操作。但是前端没有完全闭环，比如幂等建的生成规则（可以使用md5(formdata)），后端在针对nats消费者断网重传的问题并没有封装统一的幂等函数，造成代码冗余
- 目前系统中通过封装 `BoundedPublisherService` 实现了异常消息推送至死信队列中的逻辑。但是为了严密执行逻辑。还需要下面逻辑的闭环
  - 1. **自动与人工补偿机制的分级处理**：
       - **对于普通聊天消息**：业务校验在跨越 MQ 写屏障前已完成。一旦进入 MQ，对用户即为“发送成功”。若后续异步落库 DB 失败，纯属**系统基础设施故障**，只需进入 DLQ 等待运维修复 DB 后直接重放（Replay）落库，**绝对不向客户端下发失败通知**。
       - **对于高频状态/资金业务（如抢红包）**：采用“Redis 内存极速预扣 + 异步 MQ 落库”。若由于严重的底层账本校验失败导致拒绝落库，必须由补偿 Worker 自动执行内存冲正，并向客户端下发 `EXCEPTION_ACK` 失败通知。
  - 2. **进入 DLQ 失败的兜底（假死防范）**：如果在将异常消息转存至 DLQ 时，恰好 NATS 网络断开或磁盘打满导致发送失败，**绝对不能对主队列的原消息进行 ACK**。必须让原消息保留在主队列等待，同时触发严重告警，防止消息彻底蒸发。
  - 3. **客户端乐观 UI 回滚响应的适用场景**：前端必须具备接收服务端异步 `EXCEPTION_ACK` 并执行状态回退的能力（针对上述的高频业务冲正）。但这**不适用于已通过写屏障的普通聊天消息**。普通消息的乐观 UI 只在收到同步限流/拒发异常，或超时未收到 `MSG_UP_ACK` 时，才立即显示 ❗️ 红叹号。
  - 4. **7 天留存约束与实时监控告警**：基于 NATS JetStream `StorageType.File` 配置 `max_age: 7d` 提供物理防丢底线。必须在 DLQ 处挂载 Webhook 告警，一旦发生死信必须立即触发通知（要求人工在短时间内介入修复重放），“7 天留存”仅作为最终证据保存的安全网。
- 后端增加异常分类，控制前端是否执行刷新操作。
- 虚拟表格，前端性能，后端分页的优化。站在V8的角度优化代码，使用devtools优化性能。
