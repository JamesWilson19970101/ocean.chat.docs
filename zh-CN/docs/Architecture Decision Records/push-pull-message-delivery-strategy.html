ADR: 消息投递的统一推拉结合策略

状态 (Status)​

已接受 (Accepted)

日期 (Date)​

2026-05-07

背景 (Context)​

决定如何向客户端（在线和离线）投递消息。

最初，计划考虑采用“基于 Payload（有效载荷）的动态路由”或者“大小包分流”策略：后台判断如果同步的消息少于 10 条，或者少于 1KB 的时候，仅使用长连接推（Push）即可；如果不符合该条件，则使用长短连接配合同步增量消息。

从直觉上看，既然只有几条纯文本（几百字节），直接顺着长连接通道“顺手”塞给客户端，省去了 HTTP 握手的开销（RTT），用户的体感延迟确实会更低。很多早期的 IM 系统或者对实时性要求极高但对可靠性要求稍低的场景（比如直播间弹幕、游戏内公屏聊天），确实是这么做的。

但是，在类似微信、飞书这样追求“绝对不丢”和“海量并发”的国民级 IM 架构中，架构师必须仔细权衡这种看似巧妙的“条件分支”方案所带来的隐患。

决策 (Decision)​

拒绝基于 Payload 的动态路由。目前计划无论是 online message 还是 offline message，都使用统一的推拉结合方式（长连接轻量通知 + HTTP 短连接拉取实体）来保证消息的一致性和可靠性，不再根据消息大小或数量进行分流。

考虑过的替代方案 (Alternatives Considered)​

基于 Payload 的动态路由（小包 Push，大包 Pull）​

  • 优点 (Pros): 省去了 HTTP 握手的开销（RTT），对于小体积纯文本消息，用户的体感延迟更低。

  • 缺点 (Cons): 在超大规模的分布式系统中，这种做法会引入几个非常致命的架构隐患：

    1. 致命伤一：破坏了网关的“绝对无状态”。长连接网关（WebSocket/TCP Gateway）能扛住千万级并发的核心秘诀是“无脑”。它只负责维护网络通道，不关心业务。如果采纳该策略，网关或者推送服务就必须做一件事：拆包看内容。它需要去数“这里有几条消息？”，去算“这几条消息加起来有没有超过 1KB？”。一旦网关开始关心业务数据的长度和数量，它就和业务逻辑深度耦合了。CPU 消耗会成倍增加，原本一台机器能扛 100 万连接，现在可能只能扛 10 万了。
    2. 致命伤二：送达确认（ACK）的噩梦。IM 系统最难的是“怎么证明你收到了”。纯 Pull（拉取）模型极其优雅：客户端请求 Sync(seq=100)，服务器下发了 101-105。只要下次客户端请求时带上 Sync(seq=105)，服务器就确信：“哦，前面的他都收到了”，不需要额外的确认包。如果是长连接直接 Push 数据：服务器把 101-105 直接推下去，万一用户的手机刚进电梯网络断了，数据死在半路怎么办？为了保证不丢，长连接必须引入一套复杂的 ACK（回执）机制和超时重传机制。这不仅让长连接变得无比沉重，而且把网络层和业务层搅在了一起。
    3. 致命伤三：协议分叉导致的“状态机爆炸”。对于程序员来说，维护一套统一的逻辑永远比维护两套条件分支要安全得多。如果引入了该策略，一条消息的送达就有了两条完全不同的路径：
      • 场景 A： 用户在线，且消息 < 1KB ➔ 走长连接 Push。
      • 场景 B： 用户离线，或者消息 > 1KB ➔ 走 HTTP Pull。
      • 场景 C（最可怕的边界情况）： 用户刚好处于“弱网环境”。服务器长连接 Push 了一半，TCP 断了；同时客户端切到了 4G 满血复活，立刻发起 HTTP Pull。这时候服务器是该重传 Push，还是响应 Pull？数据非常容易发生重复、错乱或死锁。
    4. 致命伤四：无法抵御“群聊并发风暴”。假设你在一个 500 人的活跃大群里。大家在抢红包，瞬间产生了 10 条短消息。按照该策略，这 10 条消息都很小（< 1KB），所以服务器通过长连接同时向这 500 个人 Push 数据。那一瞬间，服务器的长连接集群要向外发送 5,000 次真正的数据包，带宽和 CPU 会瞬间飙升。而如果是“轻量通知 + 客户端分散拉取（Pull）”，配合 HTTP CDN 和缓存，压力会被打散，系统会平稳得多。

  • 拒绝原因 (Rejected): 在追求极高可靠性与海量并发的 IM 架构中，其带来的架构隐患和维护成本远大于其所节省的微小延迟。

后果 (Consequences)​

  • WebSocket/TCP 网关保持绝对的无状态和轻量化。
  • 极大简化了客户端的状态机逻辑：收到轻量通知后统一发起数据拉取。
  • 彻底消除了在长连接层实现复杂的业务级 ACK 和超时重传机制的必要性。
  • 通过 HTTP CDN、API 网关和 Redis 缓存来吸收拉取请求，提升了系统应对群聊并发风暴的韧性。
  • 虽然对于极小体积消息增加了一次 HTTP RTT 的开销，但这是换取系统绝对可靠和水平扩展能力的必要妥协。
