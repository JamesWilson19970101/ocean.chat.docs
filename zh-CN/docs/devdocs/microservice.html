微服务架构

整个平台被划分为四个逻辑层，共包含 10 个核心微服务 和 1 个数据处理管道，确保了职责的分离。

第一层：网关与接入层​

这一层是用户的直接入口，专注于处理海量并发连接，是整个系统的性能瓶颈所在。

1. 连接网关服务（有状态）​

鉴于本服务是有状态的，所以在设计本服务的过程中尽量保持其业务无关化、轻量化、简单化。

  • 核心职责:

    • 协议处理: 维护客户端的 WebSocket/TCP 长连接，处理心跳检测、连接建立与断开。
    • 数据透传: 作为纯粹的连接通道，它不处理任何业务逻辑，仅将客户端的原始数据包进行封装（如附加上 connectionId, gatewayId 等元数据），然后快速投递给后端的“消息路由服务”。
    • 客户端消息下发: 接收来自“实时推送工作单元”的指令，将消息准确地推送给连接在本实例上的客户端。

  • 分离原因: 将最消耗资源的 I/O 密集型任务（维护连接）与 CPU 密集型任务（业务逻辑）彻底分离。这使得连接网关可以被极致优化，并独立进行水平扩展，以支撑千万甚至亿级的并发连接。

2. 消息路由服务（无状态）​

  • 核心职责:

    • 消息解码与分发: 接收来自“连接网关”的原始数据包，模仿 DDP 协议，进行解码、协议解析和初步验证。
    • 业务路由: 根据消息的类型或指令，判断其应由哪个业务微服务处理，然后通过消息队列 NATS 将其分发到对应的业务服务。
    • 上行流量控制: 实现通用的速率限制和熔断机制。此处限制“每个用户 ID，每秒最多只能向后端转发 100 个请求”，但是具体的业务中比如发送普通消息的频率可以很高（比如每秒 10 条），但创建一个群组的频率就必须很低（比如每分钟 1 次）；或者根据依赖关系而进行服务熔断保护的机制应该在具体的服务中实现。

  • 分离原因: 解耦接入层和业务逻辑层。路由服务作为中间协调者，使得后端业务服务的增减和变更对网关层完全透明，极大地提高了系统的灵活性和可维护性。

第二层：核心业务逻辑层​

这一层负责处理 IM 平台所有的核心业务功能，设计为无状态服务，易于水平扩展。

3. 用户关系服务（无状态）​

  • 核心职责: 管理用户账户、个人资料、好友关系（添加/删除/黑名单）、通讯录等。
  • 分离原因: 用户和关系是 IM 的基础数据，将其独立出来，可以为其他服务提供统一、稳定的数据源。

4. 群组服务（无状态）​

  • 核心职责: 负责群组的全生命周期管理，包括群创建/解散、成员管理、权限体系、群公告、群设置等所有与群相关的功能。
  • 分离原因: 群聊的业务逻辑（尤其是权限和成员管理）非常复杂，独立成服务有助于降低代码的复杂度，便于独立开发和迭代。

5. 消息逻辑服务（无状态）​

  • 核心职责:

    • 消息处理: 这是单聊和群聊消息的业务处理中心。负责消息的权限校验（如是否是好友、是否在群内）、内容处理（@提及、敏感词过滤）、生成消息 ID、组装消息体等。
    • 触发投递: 在消息处理完成后，调用“推送编排服务”，启动消息的投递流程。

  • 分离原因: 将消息本身的业务逻辑（“是什么”）与消息的投递过程（“怎么送”）分开，使职责更清晰。

第三层：消息推送管道​

这是确保消息可靠、实时送达的关键，也是一个高度异步化的处理流程。

6. 推送编排服务（无状态）​

  • 核心职责:

    • 消息-投递决策: 接收来自“消息逻辑服务”的待投递消息。
    • 状态查询: 实时查询“在线状态服务”，获取所有接收者的在线状态和所在网关节点。
    • 任务派发: 根据在线状态，将消息拆分为“在线推送任务”和“离线推送任务”，并将这些任务原子化地写入到不同的消息队列 Kafka 中。

  • 分离原因: 作为消息投递的“大脑”，它负责复杂的决策逻辑，将其独立可以使推送流程更清晰、更易于监控和调试。

7. 实时推送工作单元（无状态）​

  • 核心职责: 监听“在线推送”队列，消费任务，然后直接与目标用户所在的“连接网关”实例通信，指令其下发消息。
  • 分离原因: 专职处理在线消息的推送，可以根据在线用户的数量和消息量进行独立的扩缩容，确保实时性。
  • 技术栈: Kafka 消费者, ioredis (用于网关间 Pub/Sub)。

8. 离线推送工作单元（无状态）​

  • 核心职责: 监听“离线推送”队列，消费任务，然后调用苹果 APNS、谷歌 FCM 或国内厂商的推送 API，发送离线通知。
  • 分离原因: 与第三方 API 的集成通常伴随着网络延迟和不确定性。将其隔离可以防止其失败或缓慢影响到核心的实时推送链路。

第四层：基础支撑服务​

这些服务为整个平台提供稳定、高效的基础能力。

9. 在线状态服务（无状态）​

  • 核心职责:
    • 状态维护: 通过 userId -> gatewayId, status 的映射，实时维护全局用户的在线状态。
    • 状态查询: 为“推送编排服务”等提供毫秒级的在线状态查询接口。
  • 分离原因: 在线状态是分布式 IM 的基石，读写极为频繁。独立服务使用 Redis 等内存数据库进行极致优化，确保高性能。

10. 数据查询服务（无状态）​

  • 核心职责:
    • 统一查询入口: 为客户端提供查询历史消息、会话列表等数据的统一 HTTP API。
    • 分级查询: 根据查询的时间范围，智能地从不同的存储介质（Redis 缓存、MongoDB 等）中拉取数据并聚合。
  • 分离原因: 实现了读写分离。将高频的读操作与核心的写链路分开，可以独立优化查询性能，而不会影响消息写入的稳定性。

数据处理管道：消息持久化​

这不是一个服务，而是一个异步处理流程。

  • 核心职责: “消息逻辑服务”在处理完消息后，除了调用推送服务，还会将消息副本发送到一个专用于持久化的消息队列（如 Kafka）。一个或多个独立的**消费者进程（Writer）**会监听此队列，批量将消息写入数据库。

分离原因: 彻底的异步化。消息的发送和接收不应等待数据库写入完成。这种“写后持久化”的设计，最大程度地保证了消息的实时性。

demo​

  1. 用户 A 发送一条消息给群 G：
  2. 客户端 A -> 连接网关-1 (建立连接)
  3. 连接网关-1 -> 消息路由服务 (透传)
  4. 消息路由服务 -> 消息逻辑服务 (路由)
  5. 消息逻辑服务 处理消息（校验权限、生成 ID），然后兵分两路：
    1. -> 推送编排服务 (启动投递)
    2. -> Kafka 持久化队列 (准备存储)
  6. 推送编排服务查询在线状态服务，得知群成员 B 在线（网关 2），C 离线。
  7. 推送编排服务 -> 在线推送队列 (任务 for B) 和 离线推送队列 (任务 for C)。
  8. 实时推送工作单元 消费任务 B -> 指令 连接网关-2 -> 客户端 B 收到消息。
  9. 离线推送工作单元 消费任务 C -> 调用 APNS/FCM API -> 客户端 C 收到通知。
  10. 持久化 Writer 消费 Kafka 消息 -> 写入 MongoDB。
